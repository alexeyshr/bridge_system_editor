# Lightweight Spec Flow

Облегченный процесс для реализации новых фич без лишней бюрократии.

## Когда нужен spec

Spec обязателен, если изменение:
- меняет бизнес-логику торговли;
- затрагивает структуру БД, API или импорт/экспорт;
- содержит сложный UI-flow;
- занимает больше 1-2 часов разработки.

Для мелких правок UI/текста spec можно пропустить.

## Структура

Для каждой фичи создается папка:

`docs/specs/YYYY-MM-DD-short-name/`

Внутри 3 файла:
- `spec.md` — что и зачем делаем;
- `plan.md` — как будем реализовывать;
- `tasks.md` — список конкретных шагов.

Шаблоны лежат в:

`docs/specs/_templates/`

Можно создать заготовку автоматически:

`powershell -ExecutionPolicy Bypass -File scripts/new-spec.ps1 -Slug <short-name>`

## Пошаговый процесс

1. Создать ветку:
   - `git checkout -b codex/<short-name>`
2. Создать папку spec и скопировать шаблоны.
3. Заполнить `spec.md` (требования + критерии приемки).
4. Заполнить `plan.md` (архитектурный подход + риски).
5. Разбить работу в `tasks.md` на маленькие проверяемые шаги.
6. Реализовать задачу, отмечая выполненные пункты в `tasks.md`.
7. Открыть PR:
   - приложить ссылку на папку spec;
   - дождаться зеленого CI;
   - только потом merge.

## Definition of Done

Фича считается завершенной, если:
- критерии приемки из `spec.md` выполнены;
- все чекбоксы в `tasks.md` закрыты;
- CI зеленый;
- изменения задокументированы в `readme.md` (если нужно).
