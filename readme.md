# K7 Editor

Локальный редактор торговой системы Бриджа (K7) с иерархическим деревом секвенций, карточкой редактирования и импортом/экспортом YAML.

## 1. Назначение проекта

Проект решает задачу ведения системы торговли в виде дерева секвенций:
- просмотр иерархии заявок с разворачиванием/сворачиванием;
- редактирование смысловых полей заявки (`shows`, `shape`, HCP, forcing, notes и т.д.);
- workflow согласования через комментарии и флаг "Принято";
- локальный импорт и экспорт YAML;
- локальный семантический поиск (без внешних API).

## 2. Что умеет сейчас

- Дерево секвенций с:
  - отступами по глубине;
  - сворачиванием веток;
  - счетчиком потомков рядом со стрелкой;
  - сортировкой заявок по bridge-порядку (`C, D, H, S, NT`).
- Фильтры:
  - все;
  - принято;
  - не принято;
  - только с комментариями.
- Фильтр глубины (`Все`, `1`, `2`, `3`, ...).
- Карточка секвенции справа:
  - редактирование shows/описания/HCP/forcing/type/alert/shape;
  - добавление комментариев;
  - изменение статуса принятия;
  - создание продолжения;
  - переименование `call` (с обновлением всей ветки);
  - удаление секвенции (с потомками).
- Импорт YAML:
  - единая кнопка `Импорт YAML` с авто-выбором источника.
- Экспорт YAML:
  - выдача `k7-standard-export.yaml`.
- Поиск:
  - локальный TF-IDF + LSA ранжировщик;
  - fallback на `LIKE`-поиск, если семантика недоступна/слабая.

## 3. Текущая архитектура (as-is)

Проект сейчас реализован как монолит "backend + frontend в одном файле":

- Backend: FastAPI.
- Storage: SQLite (`k7.db`).
- Frontend: встроенный HTML/CSS/JS как строка `HTML` внутри `main.py`.
- API и UI находятся в одном процессе.

### 3.1 Слои

1. `Data layer` (SQLite):
   - таблицы `sequences`, `status`, `comments`, `revisions`;
   - основная иерархия хранится через `seq_key` + `parent_key`.
2. `Application layer` (Python/FastAPI):
   - импорт, экспорт, CRUD секвенций, комментарии, статус;
   - операции переименования/удаления веток;
   - семантический индекс поиска.
3. `Presentation layer` (чистый JS):
   - получает JSON через `fetch`;
   - рендерит таблицу и карточку;
   - управляет фильтрами, выбранной строкой, модалками.

### 3.2 Ключевая идея хранения дерева

- `seq_key` хранит полный путь секвенции, например: `1C 1D 1H 2H`.
- `parent_key` хранит родителя: `1C 1D 1H`.
- Это позволяет:
  - быстро строить дерево на клиенте;
  - делать операции над поддеревом через `WHERE seq_key = ? OR seq_key LIKE ?`.

## 4. Структура файлов

- `main.py`:
  - вся бизнес-логика, API и встроенный фронтенд.
- `requirements.txt`:
  - зависимости Python.
- `k7.db`:
  - рабочая база данных.
- `k7-export.yaml`:
  - временный файл экспорта (создается при `/api/export`).
- `../YAML_system/*.md`:
  - локальный источник секвенций в markdown с YAML-блоками.

## 5. База данных

### 5.1 Таблица `sequences`

Основные поля:
- `seq_key` (UNIQUE) — полный путь секвенции.
- `parent_key` — родительский путь.
- `call` — текущая заявка.
- `meaning_json` — исходный смысловой JSON.
- `hcp_min`, `hcp_max`, `forcing`, `seq_type`, `notes`, `alert`, `side`, `section`.

### 5.2 Таблица `status`

- `seq_key` (PRIMARY KEY), `accepted`, `updated_at`.
- Хранит статус принятия независимо от основного контента.

### 5.3 Таблица `comments`

- `seq_key`, `author`, `text`, `created_at`.
- По умолчанию автор: `Алексей`.

### 5.4 Таблица `revisions`

- История изменений полей (`field`, `old_value`, `new_value`, `changed_at`).

## 6. Источники данных и импорт

Единый endpoint `POST /api/import` выбирает источник:

1. Если существует `YAML_SRC` (`k7-standard.yaml`) — импортирует его.
2. Иначе, если есть папка `YAML_system` — импортирует markdown/YAML оттуда.
3. Если источников нет — HTTP 400.

### Важный момент

При импорте очищается `sequences`, а `status/comments/revisions` не очищаются автоматически. Это позволяет сохранять процесс согласования, но может оставлять "осиротевшие" записи для удаленных ключей.

## 7. Поиск

### 7.1 Семантический режим

Локально строится индекс:
- word n-gram TF-IDF (1..2);
- char n-gram TF-IDF (3..5);
- TruncatedSVD (LSA) при достаточной размерности.

Запрос ранжируется по cosine similarity + небольшой lexical boost.

### 7.2 Fallback

Если семантический ранжировщик недоступен или top score слишком низкий, выполняется обычный SQL `LIKE` по:
- `seq_key`;
- `notes`;
- `forcing`.

## 8. API (основное)

- `GET /api/sequences`
  - список секвенций с фильтрами `section`, `search`, `accepted`, `commented`.
- `GET /api/sequences/{seq_key}`
  - карточка секвенции + комментарии + прямые дети.
- `PUT /api/sequences/{seq_key}`
  - обновление полей секвенции.
- `POST /api/sequences`
  - создание новой секвенции.
- `PUT /api/seq-rename/{seq_key}`
  - переименование `call` у узла + пересчет ключей/родителей всего поддерева.
- `DELETE /api/seq-delete/{seq_key}`
  - удаление секвенции и всех потомков.
- `POST /api/sequences/{seq_key}/comment`
  - добавление комментария и автоматический сброс `accepted=0`.
- `PUT /api/seq-status/{seq_key}`
  - изменение статуса принятия; при `accepted=true` комментарии по узлу удаляются.
- `GET /api/stats`
  - агрегаты для шапки/вкладок.
- `POST /api/import`
  - импорт из YAML.
- `GET /api/export`
  - экспорт в YAML-файл.

## 9. Логика UI

### 9.1 Таблица

- Строка может быть:
  - обычная;
  - selected (`sel`);
  - accepted (`acc`);
  - merged (`row-merged`, если в `seq_key` есть `/`).
- Слева:
  - `✅` если принято;
  - `💬` если есть комментарии и не принято;
  - `·` иначе.
- Рядом со стрелкой:
  - счетчик потомков для узлов с детьми.

### 9.2 Карточка справа

- Показывается по выбору строки.
- Позволяет:
  - править смысловые поля;
  - добавлять комментарии;
  - создавать продолжения;
  - переименовывать/удалять узел.

### 9.3 Создание новой секвенции

После создания UI:
1. сбрасывает поиск/фильтры/секцию/глубину на `Все`;
2. перезагружает список;
3. выбирает новую строку;
4. открывает карточку справа.

## 10. Переименование и удаление ветки

### 10.1 Переименование

`PUT /api/seq-rename/{seq_key}`:
- проверяет конфликты ключей;
- временно переносит поддерево в служебный namespace (чтобы обойти `UNIQUE`);
- записывает финальные `seq_key`/`parent_key`/`call`/`section`;
- переносит связанные записи в `status/comments/revisions`.

### 10.2 Удаление

`DELETE /api/seq-delete/{seq_key}`:
- удаляет выбранный узел и все его продолжения;
- очищает связанные строки из `comments/status/revisions`.

## 11. Запуск локально

Из папки `k7-editor`:

```bash
python -m venv .venv
.venv\\Scripts\\activate
pip install -r requirements.txt
python main.py
```

Или через uvicorn:

```bash
uvicorn main:app --host 127.0.0.1 --port 8001 --reload
```

Открыть в браузере:
- `http://127.0.0.1:8001/`

Проверочная страница:
- `http://127.0.0.1:8001/test`

## 12. CI (GitHub Actions)

В репозитории настроен базовый workflow:
- файл: `.github/workflows/ci.yml`;
- запуск: при `push` в `main` и `codex/**`, а также для `pull_request` в `main`.

Проверки в CI:
- `ruff` (критические правила: синтаксис/фатальные ошибки);
- `python -m py_compile main.py` (проверка компиляции);
- smoke-check импорта (`import main`, наличие `app`).

Цель этого CI:
- быстро ловить поломки до merge в `main`;
- дать единый автоматический quality gate для изменений.

## 14. Технические ограничения (текущее состояние)

- Архитектура монолитная, один большой `main.py` (сложно масштабировать и тестировать).
- Нет аутентификации и ролей.
- Нет миграций схемы БД (ручная эволюция структуры).
- Импорт полностью перезаписывает `sequences`.
- `status/comments/revisions` могут содержать старые ключи после импорта.
- Frontend без модульной сборки/типизации.

## 15. Направление эволюции

Практичный путь развития:
1. Разделить backend и frontend по файлам.
2. Вынести работу с БД в отдельный слой репозиториев.
3. Добавить миграции (`alembic` или lightweight migration runner).
4. Добавить чистку orphan-записей после импорта.
5. Ввести версионирование/снапшоты системы.
6. Добавить пользователей и авторство комментариев.
